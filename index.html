Arquitectura del Sistema

Frontend

Aplicación web para registrar clientes, ver deudas y generar informes. Tecnologías recomendadas:

  • Framework: React
  • Manejo de estado: Context API/Redux (React)
  • UI:Tailwind CSS

Backend

API para manejar los datos de clientes, deudas y tasas de cambio:

  • Framework: ASP.NET Core Web API
  • ORM: Entity Framework Core
  • Autenticación: JWT (si es necesaria)
  • Consumo API tasas: HttpClient para obtener tasas del dólar

Base de Datos

Almacenamiento de clientes, deudas y tasas del dólar:

  • Recomendada: SQL Server o PostgreSQL
  • Alternativa: SQLite para entornos ligeros
  • Modelo: Tablas relacionadas (Clientes → Deudas)

Funcionalidades Clave

Gestión de Clientes

  • Crear, editar y eliminar clientes
  • Búsqueda y filtrado de clientes
  • Visualización de historial

Gestión de Deudas

  • Registrar deudas con monto y fecha
  • Asociar tasa de cambio del día
  • Marcar deudas como pagadas
  • Visualizar deudas por cliente

Informes

  • Reporte semanal de deudas
  • Histórico de tasas de cambio
  • Exportar a PDF/enviar por email

Conceptos Básicos de Bases de Datos

storage Bases de Datos SQL (Relacionales)

  • Estructura: Datos organizados en tablas con filas y columnas
  • Ventajas: Integridad de datos, ACID, relaciones complejas
  • Ejemplos: PostgreSQL, SQL Server, MySQL
  • Uso ideal: Aplicaciones transaccionales, datos estructurados

Ejemplo SQL:

CREATE TABLE deudas (
    id SERIAL PRIMARY KEY,
    cliente_id INT REFERENCES clientes(id),
    monto DECIMAL(10,2),
    fecha DATE,
    tasa DECIMAL(10,2),
    pagada BOOLEAN DEFAULT false
);

data_object Bases de Datos NoSQL

  • Estructura: Documentos, clave-valor, grafos o columnas
  • Ventajas: Escalabilidad horizontal, flexibilidad
  • Ejemplos: MongoDB, Cassandra, Redis
  • Uso ideal: Big Data, datos no estructurados

Ejemplo NoSQL (MongoDB):

{
  "cliente": "Juan Pérez",
  "deudas": [
    {
      "monto": 150.50,
      "fecha": "2023-10-15",
      "tasa": 35.20
    }
  ]
}

schema Modelado de Datos para este Proyecto

Estructura recomendada para el sistema de gestión de deudas:

  • Clientes: id, nombre, contacto, dirección
  • Deudas: id, cliente_id, monto, fecha, tasa, estado
  • Tasas: id, fecha, valor, fuente
  • Relaciones: Un cliente tiene muchas deudas

Consulta JOIN típica:

SELECT c.nombre, d.monto, d.fecha, d.tasa
FROM clientes c
JOIN deudas d ON c.id = d.cliente_id
WHERE d.pagada = false;
